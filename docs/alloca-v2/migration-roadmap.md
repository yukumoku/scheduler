# Alloca v2 Migration Roadmap

Version: 2.0 draft

この文書は、現行実装から Alloca v2 へ移行するための段階的な計画をまとめたものである。

## 1. 移行の基本方針

- いきなり全面置換しない
- まずデータの中心を揃える
- 次に概念を分離する
- その後に UI を整理する
- 最後にシフト生成・AI・カレンダーを積み上げる

## 2. Phase 1: 参加可能日時のイベント化

### 目的

参加可能日時を、イベント単位で扱える唯一の入力源として確立する。

### 実施内容

- `CommonAvailabilitySet` / `CommonAvailability` を `AvailabilitySet` / `Availability` の移行名として整理する
- 参加可能日時セットを `Event` 配下に移す
- 必要に応じて `team_id` で班単位のセットも作れるようにする
- イベント別希望提出導線を削除または互換レイヤー化する
- 提出状況、集計、一覧のロジックを共通 service に集約する

### 完了条件

- ユーザーは参加可能日時だけを提出できる
- 管理者は参加可能日時の提出状況を一元確認できる
- イベント別希望の UI が見えない
- 既存のイベント別 API は互換のために残っていても、新規開発では使わない

## 3. Phase 2: 班のイベント化

### 目的

`Team` を Event 配下の運用単位として整理する。

### 実施内容

- `Team` を `Event` 直下に移す
- `TeamMember` を Event 内の班メンバーとして整理する
- `leader` / `member` の役割を明確にする
- 班管理画面を Event Detail 主導にする

### 比較結果

#### Event 側に寄せる案

メリット:

- 班が「このイベントのための班」として自然になる
- 同じグループでもイベントごとに班構成を変えられる
- 文化祭・体育祭のような単位と相性がよい

デメリット:

- 既存 `Group` 直下の班実装からの移行が必要

#### Group 側に残す案

メリット:

- 既存実装からの変更量が少ない

デメリット:

- 班がイベントと結びつかず、利用者のイメージとずれる
- どのイベントの班か分かりにくい

### 結論

**Event 側に寄せる。**

理由:

- 2年C組 → 文化祭 → ロッカー班 という利用イメージに合うから
- ルールやメンバーがイベントごとに変わるから

## 4. Phase 3: Task 分離

### 目的

`EventTask` と `EventSlot` の責務を分ける。

### 実施内容

- `EventTask` を導入する
- `EventSlot` から作業概念を外す
- `target_team_id` / `allow_cross_team_help` を Task 側へ寄せる
- `EventSlot` は時間枠専用にする

### 完了条件

- 受付・装飾・撤収が作業単位として表現できる
- 時間枠は純粋に日時と必要人数だけを持つ

## 5. Phase 4: UI 整理

### 目的

スマホでも迷わない情報構造にする。

### 実施内容

- カード中心の一覧をテーブル / リストに変更する
- 重要情報を上部に集約する
- 主操作を 1 つに絞る
- サブ操作をタブやメニューに移す

### 完了条件

- 1 画面で情報の優先順位が明確
- 横スクロールが発生しない
- スマホでボタンが見切れない

## 6. Phase 5: Shift 生成

### 目的

アルゴリズムベースでシフトを生成できるようにする。

### 実施内容

- 参加可能な人だけを候補化する
- 対象班を優先する
- 不足時のみ他班へ拡張する
- 公平性と希望反映率を考慮する

### 完了条件

- 手動での調整なしに初期シフトを作れる
- 調整理由を追跡できる

## 7. Phase 6: Calendar 設計

### 目的

Alloca 内の予定を 1 つのカレンダーとして扱う。

### 実施内容

- `CalendarItem` または `CalendarItemDTO` を導入する
- 表示用データと保存用データを分ける
- Google Calendar / Outlook 連携の前提を置く

### 完了条件

- グループ・イベント・班・シフトを横断して表示できる
- 外部同期へ拡張しやすい

## 8. Phase 7: AI Shift Assistant

### 目的

AI を生成ではなく分析・提案に限定して導入する。

### 実施内容

- 不足枠の分析
- 偏りの分析
- 改善提案
- ヘルプ候補の提案
- 提案理由の説明

### 完了条件

- シフトはアルゴリズムが主担当
- AI は補助役として動く

## 9. 現行コードからの削除候補

- `EventAvailabilityController`
- `AvailabilityPage`
- イベント別希望提出 API
- イベント別希望提出 UI
- Group 直下の班管理を正とした導線
- Group 直下の参加可能日時管理を正とした導線

## 10. 残すもの

- `Group`
- `GroupMember`
- `Event`
- `Team`
- `TeamMember`
- `AvailabilitySet`
- `Availability`
- `EventTask`
- `EventSlot`
- `Shift`

## 11. 新しく追加するもの

- `AvailabilitySummaryService`
- `ShiftAssistantService`
- `CalendarItem` または `CalendarItemDTO`

## 12. 段階的な実装順序

以下の順番で実施する。

1. 参加可能日時をイベントへ寄せる
2. 班をイベントへ寄せる
3. `EventTask` を導入する
4. UI をタブとテーブル中心へ再構成する
5. シフト生成をアルゴリズム化する
6. カレンダー基盤を追加する
7. AI の分析・提案機能を追加する

## 13. 移行時の注意点

- 既存 API を急に壊さない
- 互換が必要なものは互換レイヤーで吸収する
- 一度に多くの概念を追加しない
- 先に「どれが正か」を決めてから実装する
